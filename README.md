# 📊 Crypto Market Insight

## 1. 프로젝트 소개

Crypto Market Insight는 가상자산 시장 데이터를 기반으로
체인/생태계 관점의 코인 분류, 기술적 지표 분석,
그리고 전략 백테스트 및 모의 투자를 제공하는 데이터 분석 웹 서비스입니다.

본 프로젝트는 단순한 시세 조회를 넘어,
사용자가 시장 구조를 이해하고 전략을 검증할 수 있는 환경을 제공하는 것을 목표로 합니다.

---

## 2. 문제 정의

대부분의 가상자산 차트 서비스는 다음과 같은 한계를 가지고 있습니다.

- 가격 및 기본 지표 위주의 단편적인 정보 제공
- 체인/생태계 관점의 구조적 분석 부족
- 전략의 성과를 검증하기 위한 백테스트 환경 부재
- 사용자가 직접 데이터를 수집·가공해야 하는 높은 진입 장벽

이로 인해 사용자는 `왜 이 코인이 움직였는지`,
`이 전략이 과거에도 유효했는지`를 판단하기 어렵습니다.

본 프로젝트는 이러한 문제를 해결하기 위해
`데이터 수집 → 분석 → 전략 검증 → 의사결정`으로 이어지는 흐름을 하나의 서비스로 통합합니다.

---

## 3. 핵심 기능

### 3.1 시장 및 차트 분석

- 코인 시세 및 OHLCV 차트 시각화
- 타임프레임별 가격 추세 분석
- 거래량 및 변동성 지표 제공

### 3.2 체인 / 생태계 기반 분류

- BTC, ETH(ERC-20), SOL 등 주요 체인 기준 코인 분류
- 생태계별 시장 흐름 요약 및 비교
- 코인 상세 페이지에서 소속 생태계 명확히 표시

### 3.3 기술적 지표 및 신호

- MA / EMA, RSI, MACD, Bollinger Bands 등 주요 지표 제공
- 규칙 기반 매수·매도 시그널 생성
- 지표 조합에 따른 전략 구성

### 3.4 전략 백테스트

- 선택한 전략을 기준으로 과거 데이터 시뮬레이션
- 누적 수익률, 최대 낙폭(MDD), 승률 등 결과 제공
- 거래 수수료 및 슬리피지 고려

---

## 4. 아키텍처 개요

본 프로젝트는 시계열 데이터 특성과 비용 효율성을 고려하여 다음과 같은 구조로 설계되었습니다.

- 시세 및 캔들 데이터는 외부 공개 API를 통해 요청 시 조회
- 원본 시계열 데이터는 장기 저장하지 않음
- 서버 캐싱을 통해 반복 요청 비용 최소화
- 데이터베이스에는 사용자 및 전략 관련 데이터만 저장

이를 통해 불필요한 스토리지 및 I/O 비용을 방지하면서도
실시간 분석이 가능한 구조를 유지합니다.

## 6. 프로젝트 관리 및 브랜치 전략

### GitHub Projects & Issues

- **GitHub Projects**를 활용한 작업 관리
- **Issue 기반** 작업 추적 및 PR 연동
- 이슈 템플릿을 통한 일관된 작업 정의

### 브랜치 전략 (GitHub Flow)

단순하고 빠른 배포 주기에 적합한 **GitHub Flow**를 사용합니다.

```bash
main ─────────────────────────────────────────────►
       \                     /
        └── feature/xxx ────┘
```

| 브랜치      | 용도                             |
| ----------- | -------------------------------- |
| `main`      | 배포 가능한 상태 유지            |
| `feature/*` | 기능 개발 (`feature/add-login`)  |
| `fix/*`     | 버그 수정 (`fix/chart-render`)   |
| `docs/*`    | 문서 작업 (`docs/update-readme`) |

### 작업 흐름

1. **Issue 생성** → 작업 정의
2. **브랜치 생성** → `feature/이슈번호-설명`
3. **개발 및 커밋** → 작업 진행
4. **PR 생성** → 코드 리뷰 요청
5. **Merge** → `main`에 병합

> 브랜치명 예시: `feature/12-add-backtest-api`, `fix/15-chart-loading`

## 7. 환경 설정 및 실행 방법

### 개발 환경 vs 배포 환경

| 구분      | Database                    | 비고                     |
| --------- | --------------------------- | ------------------------ |
| 개발 환경 | Docker Compose (PostgreSQL) | 로컬에서 컨테이너로 실행 |
| 배포 환경 | Supabase (PostgreSQL)       | 클라우드 매니지드 서비스 |

### 환경 변수 설정

```bash
# 백엔드 환경 변수
cp backend/.env.example backend/.env

# 프론트엔드 환경 변수
cp frontend/.env.example frontend/.env
```

> `.env.example` 파일을 복사한 후, 실제 값으로 수정하여 사용

### 개발 환경 실행 (Docker Compose)

```bash
# 개발용 DB 및 인프라 실행
docker-compose up -d

# DB 컨테이너만 실행
docker-compose up -d db
```

> 개발 시에는 Docker Compose로 PostgreSQL을 로컬에서 실행하여 사용

### 애플리케이션 실행

| 구분     | 명령어                 |
| -------- | ---------------------- |
| Backend  | `./gradlew bootRun`    |
| Frontend | `npm i && npm run dev` |

```bash
# 백엔드 실행 (backend 디렉토리에서)
cd backend
./gradlew bootRun

# 프론트엔드 실행 (frontend 디렉토리에서)
cd frontend
npm i && npm run dev
```

## 8. 데이터 및 비용 전략

가상자산 시계열 데이터는 데이터 양과 접근 빈도가 매우 높아,
무분별한 적재 시 스토리지 및 운영 비용이 급격히 증가합니다.

본 프로젝트에서는 다음과 같은 전략을 적용했습니다.

- 원본 시계열 데이터는 저장하지 않고 요청 시 조회
- 최근 데이터만 메모리 캐시로 관리
- 백테스트 결과 및 사용자 설정 정보만 영속화

이를 통해 포트폴리오 수준의 프로젝트임에도 불구하고
실제 서비스 환경을 고려한 비용 통제 설계를 적용했습니다.
